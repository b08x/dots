# Codebase Documentation Structure Guide

This guide defines the canonical structure for documentation generated by
the codebase-mapper skill. Every template uses SFL process types — Material
(what transforms), Relational (what connects), Mental (what users experience).
Read `references/sfl_writing_guide.md` for modality calibration and the
full writing guide before filling these in.

---

## Top-Level Document Structure

```
docs/
├── README.md               # Project overview (entry point)
├── architecture.md         # Component relationships + data flow + diagrams
├── modules/
│   └── <module-name>.md    # One file per top-level module/package
├── data-flow.md            # How data moves through the system (sequence diagram)
├── api-reference.md        # Public interfaces, functions, classes (optional)
└── decisions.md            # Design rationale extracted from WHY/NOTE/HACK comments
```

## Extracting Content from graphify Outputs

### From GRAPH_REPORT.md:
- **God nodes** → README "Key Concepts" table (include confidence tags)
- **Surprising connections** → README "Surprising Connections" (note confidence)
- **Suggested questions** → README "Questions This Graph Can Answer"
- **WHY/NOTE/HACK extractions** → decisions.md entries

### From wiki/ folder:
- Each wiki file maps to one `docs/modules/<name>.md`
- Supplement with transformation contract language — wiki files are descriptions,
  templates require explicit inputs/outputs/conditions

### From callflow HTML / diagrams.md:
- Architecture diagrams → architecture.md "Component Relationship Map"
- Sequence diagrams → data-flow.md "Primary Operation"
- Split diagrams if a section covers multiple subsystems

### From qmd (if indexed):

Refer to **`references/qmd_codebase_patterns.md`** for the full task-to-query-type mapping.
Use the **qmd skill** for general query syntax (lex/vec/hyde, MCP tools, CLI flags).
Quick reference for the most common doc enrichment tasks:

- **Module docs** — `vec` query: "how does [ModuleName] work and what does it produce"
  + `lex` query: "[ClassName] [primaryMethod]" → transformation contract + key components
- **decisions.md** — `lex` query: `"# WHY" OR "# NOTE" OR "# HACK"` → catches comments
  graphify may not have extracted as nodes; cross-reference with graph WHY/NOTE/HACK nodes
- **Limitations** — `hyde` query (write what a limitation comment looks like) + `lex`:
  `"TODO FIXME \"not supported\""` → Known Limitations section in module docs
- **Batch source read** — `multi_get` with glob (e.g. `"src/auth/**/*.ts"`) → read actual
  source before writing transformation contracts; faster than individual `get` calls

### Modality Decisions During Extraction:
- graphify says EXTRACTED → write with strong modality (no hedge)
- graphify says INFERRED → use "typically", "appears to", "likely"
- graphify says AMBIGUOUS → use "may", "possibly", "varies"; add verification note
- You found it in code directly → EXTRACTED, strong modality
- You're reasoning from structure → INFERRED, medium modality
